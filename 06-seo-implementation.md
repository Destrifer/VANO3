## Архитектура: хаб-модель и тематическая сетка

Это смысловой каркас, на который ложатся Этапы 3–4. Реверс-инжиниринг лидера ниши (coral-print, ~670 URL) показал работающую схему — переносим её, исправляя их ошибки.

### Ядро = матрица «продукт × оси модификации»

Под-страницы строятся не из ключей по одному, а по **осям модификации продукта**, и эти оси = **опции конфигуратора / производства** (значит, тянутся из Directus):

| Ось | Источник в Directus | Примеры значений (визитки) |
|---|---|---|
| Постпечать/отделка | `finishing_options` | ламинация (глянец/мат), фольгирование, тиснение, скругление |
| Материал | группы `papers` | дизайнерская бумага, пластик, картон |
| Форма | `sizes`/`allow_round`/`allow_complex` | круглые, сложной формы |
| Тир | (контент) | VIP/премиум |
| Аудитория | (внешняя ось) | для ресторанов, для врачей |
| Гео | (внешняя ось) | по району/метро — **только при реальном присутствии** |

Матрица `продукт × значение_оси` = список **кандидатов**, НЕ план. В страницы превращаются только ячейки, прошедшие триангуляцию + 3 фильтра (05 §49, §77). Полный декартов перебор = дорвей-спам (05 §36). Соответствие модели: `promoted_pages.предустановка_опций` — это **одна ячейка матрицы** (продукт с предвыбранной опцией + свой контент). Бонус: **sitemap конкурента — готовый валидированный список осей/значений**, сам по себе источник триангуляции.

### Сетка = дерево + латеральные связи

Каждый узел — индексируемая страница; связи плотные:

```
Категория (хаб L0)  → вниз: список услуг
   ↓
Услуга/Продукт (хаб L1)  ↑ вверх: крошки   ↓ вниз: карточки сегментов с ценой «от»
   ↓
Под-интент (L2 = promoted_page)  ↕ вбок: siblings (фольга ↔ тиснение ↔ дизайн. бумага)
   → инфо↔коммерция: гайды/блог (обе стороны)   → под-кластер: примеры работ, отзывы
```

Плотность сетки — сигнал тематической глубины и рост авторитетности без закупки ссылок (05 §33).

### Решение по URL

Иерархию несут **крошки + перелинковка**, не глубина пути (Google не требует вложенности URL). → Оставляем **плоские слаги** (текущий `[slug].astro`), хабность строим связями. Вложенные пути (`[...path].astro`) — опционально, не приоритет.

### Что значит «сделать страницу хабом» (порядок работ)

Сейчас `/vizitki` — лист. Хабом его делают три связи: **вниз** (блок карточек под-интентов с ценами «от» → `promoted_pages`), **вверх** (крошки, Этап 1), **вбок** (смежные услуги + гайды). Отсюда: каркас хаба (место под карточки сегментов) закладываем в **Этапе 3**, наполняем листьями в **Этапе 4**. Категория (`type: "category"`) — хаб L0 над продуктами.

## Этап 4 — pSEO (после базовых услуг)

Стратегия — 05 §43–65, модель данных — 04 §68–98. Здесь — как реализуем. **Запускается только после Этапов 1–3**: pSEO-страница переиспользует анатомию услуги и ссылается на родителя, поэтому базовый шаблон должен существовать раньше (совпадает с «посевным запуском», 05 §81).

### Общий паттерн (контекст)

Шаблон + набор данных → много URL: одна раскладка, N строк-данных, каждая строка = отдельная страница со своим URL/мета/H1/контентом и **self-canonical**. Два режима генерации: **build-time SSG** (на сборке HTML на строку — наш случай) и on-demand ISR (для миллионов волатильных страниц — избыточно). Граница «pSEO vs спам» (05 §49) — не объём автоматики, а **перебор без пользы**: генерим из явного `published`-списка, не из декартова произведения опций. Три фильтра на строку: спрос + уникальный контент + коммерческий интент.

### Предусловия

1. ~~Конфигуратор не принимает пресет и не читает URL~~ — ✅ **СДЕЛАНО.** Логика в `src/composables/calcUrlState.ts`: `applyPreset(calc, preset)` (семантический пресет `{shape,sides,quantity,paperIndex,foil,foilColorIndex,laminationIndex}` с защитой диапазонов), `presetFromSearch`/`searchFromCalc` (короткие читаемые параметры `?foil=1&paper=…&qty=…`), `applyPresetFromUrl` (один раз читает URL на клиенте), `buildShareUrl` (собирает ссылку на расчёт). Подключено: `ProductConfigurator` принимает проп `preset` → `SheetConfigurator` применяет в setup (работает в SSR-рендере острова) + `applyPresetFromUrl` в `onMounted`. **Важно по SEO:** URL **НЕ** синхронизируется автоматически — параметры появляются только по кнопке «Получить ссылку на расчёт» (`OrderPlate`-зона), чтобы в обычном обходе не плодились параметрические URL. Кластерный пресет приходит с сервера по **чистому** URL. Проверено: смена опций не меняет адрес; кнопка копирует `…?foil=1&lam=2`; расшаренная ссылка открывает конфигуратор с пресетом. ⚠️ Реализовано для **листового** конфигуратора; многостраничный (`MultipageConfigurator`) пресет пока не принимает (нужно при кластерах на multipage-продуктах).
2. **Базовый шаблон услуги (Этап 3) готов и наполнен** хотя бы на флагмане — pSEO-страница его расширяет. ✅ визитки наполнены (Ф2).

### Реализация по шагам

1. **`src/lib/promotedPages.ts`** (новый) — `getPromotedPages()` (фильтр `status=published`, все поля) и `getPromotedPage(slug)`. По образцу `products.ts`/`works.ts`, graceful fallback.
2. **Маршрут — в тот же `src/pages/[slug].astro`.** Сейчас `getStaticPaths` отдаёт `category` + `product`; добавляем третий тип `promoted`:
   ```
   getStaticPaths → [...categoryPaths, ...productPaths, ...promotedPaths]
   ```
   Это правильнее отдельного роута: продвигаемая страница — вариант страницы услуги (тот же конфигуратор, та же анатомия), со своим контентом и пресетом. Slug'и плоские и общие → **гарантия уникальности** (валидация в Directus) + порядок приоритета на случай коллизии.
3. **Ветка `type === "promoted"`** рендерит ту же анатомию услуги, но:
   - H1/title/meta — из полей самой страницы (`h1`, `meta_title`, `meta_description`);
   - `intro_text`, `галерея`, `faq` — свои;
   - **`canonical` на саму себя**, НЕ на родителя (05 §60 — иначе самодедупликация);
   - конфигуратор открывается с `предустановка_опций` (через предусловие 1);
   - внутренние ссылки родитель ↔ под-вариант («Визитки» ↔ «Визитки с фольгированием») — иерархия + анти-каннибализация (04 §97).
4. **JSON-LD/мета** — тот же набор, что у услуги (`Product/Offer`, `BreadcrumbList`, `FAQPage`), но из полей promoted-страницы.
5. **Sitemap** подхватит автоматически — это обычные статические маршруты.
6. **Контент** — процесс 05 §88: ИИ оформляет реальные факты владельца, ручная модерация фактов. Нечем наполнить (своя галерея/специфика/FAQ) — страницы нет.

### Уровни Tier-1 / Tier-2 (плитка vs страница без плитки)

Кластерная страница (`promoted_pages`) может быть двух уровней — разделяет **одно поле, без хардкода слагов в коде**:

- **`show_as_tile` (boolean, default `false`)** в `promoted_pages`. `true` → **Tier-1**: карточка-плитка на хабе/сиблингах. `false`/null → **Tier-2**: страница есть (роут + sitemap), но не плитка, а текстовая ссылка в блоке «Смотрите также».
- Код читает поле в `promotedPages.ts` (`showAsTile: p.show_as_tile === true` — graceful: нет поля → Tier-2). Деление в **двух местах**: `src/pages/[slug].astro` (плитки хаба) и `src/pages/[product]/[cluster].astro` (сиблинги) — `filter(showAsTile)` → `SegmentTiles`, остальное → `RelatedLinks.astro`.
- Дефолт `false` = **новая страница по умолчанию Tier-2** (в плитки промотируем явно). Существующие 7 кластеров забэкфиллены в `true`.
- Tier-2 запускать **пачкой** (визитки: ламинация + УФ-лак + шелкография), иначе блок «Смотрите также» из одной ссылки выглядит сиротливо. Отбор уровня — Шаг 4.5 PLAYBOOK (Tier-разметка ядра).

### Анти-каннибализация (на этапе отбора, не кода)

Сгруппировать синонимы в один интент **до** подсчёта страниц («фольгирование» = «тиснение фольгой» = «золотой/серебряной фольгой» → одна страница, формулировки внутри в секциях/FAQ), 05 §79. Объём найденных формулировок ≠ объём страниц.

- инфо-блог (`/blog/chto-takoe-*`) со ссылками в коммерческие страницы;
- под-кластер отзывов по продуктам (`/reviews/businessCards`);
- гигиена `robots.txt` (Disallow пагинации/поиска/корзины, `Clean-param` для Яндекса, конструктор вне индекса).

**Делаем лучше (их дыры — наша фора):**
- у них **нет JSON-LD нигде** (ни Product/Offer, ни LocalBusiness, ни BreadcrumbList) при готовых крошках/ценах/3 локациях/отзывах → наш Этап 1 даёт rich-сниппеты, за которые они не борются;
- нет meta description и OG на проверенных страницах;
- **нет FAQ-блоков нигде** → наш FAQ ловит вопросительный хвост и идёт в AI-ответы (фора уже **не** про rich-сниппет — его Google убрал с мая 2026 — а про покрытие в AEO/Яндексе и поведенческие);
- в sitemap нет `lastmod`.

**Избегаем:**
- тонкие **гео-дорвеи** по метро/городам без реального присутствия (`…/vizitki-belorusskaia`, `…-v-lipecke`) — уникален ~1 абзац, остальное шаблон; анти-паттерн 05 §37. Гео — только при настоящей дифференциации.

### Механизмы пресета конфигуратора (для кластеров)

Кластер открывает конфигуратор с предустановкой. Поля пресета (`promoted_pages.preset`, JSON) по типам:

- **sheet** (`applyPreset`, `calcUrlState.ts`): `shape, sides, sizeIndex, foldTypeIndex, paperIndex, foil, foilColorIndex, laminationIndex, finishing[]` (доп.отделка по имени), `quantity`.
- **multipage** (`applyMultipagePreset`): `formatIndex, bindingId, pages` (+ `quantity, foil, laminationIndex`). Пробрасывается в `MultipageConfigurator`.
- **fixed** (3D/смола): своего пресета пока нет; `fixed_price` (₽/лист по числу листов) + `fixed_sheet_*`.

> **Урок миграции:** модификаторы, заведённые отдельными продуктами, переводим в `promoted_pages`‑кластеры; старые продукты снимаем с публикации (`status=draft`). Редиректы — только если URL были в индексе.

> 🐛 **Баг гидрации (исправлен 2026-06-15):** `PaperSelect.vue` биндил `<select :value>+@change` — Vue **терял выбор бумаги при гидрации** острова, из‑за чего серверный пресет `paperIndex` сбрасывался в 0 во всех разделах. Перевели на `v-model` (computed‑прокси). Урок: **новые `<select>` в островах — только через `v-model`**, иначе SSR‑значение не доживает до клиента.
>
> 🐛 **Тот же баг в `FormatField.vue` (исправлен 2026-06-16):** селект формата тоже был на `:value+@change` → пресет **`formatIndex` молча сбрасывался в 0** во ВСЕХ multipage‑кластерах с пресетом формата (брошюры `/brochures/a4`,`/a6`; журналы `/magazines/newspapers`=A3). Переведён на `v-model` через computed‑прокси `indexProxy`. Проверено: A3/A6 переживают гидрацию. **Вывод: проверять пресет формата в превью ПОСЛЕ гидрации, а не только в пропсах острова.**
