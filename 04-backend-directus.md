# 04 · Бэкенд и модель данных (Directus)

## Почему Directus

Нужен не «контентный CMS» для статей, а **слой данных с админкой поверх базы**. Directus:
- database-first: оборачивает SQL-базу (Postgres/MySQL/SQLite) headless-слоем, изменения схемы создают настоящие колонки, данные переносимы (не заперты в проприетарном формате);
- даёт автогенерируемую админку + REST/GraphQL — без ручного SQL и самописного бэкенда;
- не привязан к Next.js (в отличие от Payload) — чисто живёт рядом с Astro;
- **ревизии (Revisions)** — версионирование с откатом через интерфейс. Критично для типографии, где цены меняются и ошибаются: откатить ошибочную правку прайса — встроенная возможность.

(Альтернативы рассмотрены: Strapi — устаревшая админка, слабее TS, ограничительнее лицензия; Payload — отличный, но сросся с Next.js; Supabase — мощно, но студия для разработчика, не клиентская админка. Возможна связка Directus поверх Postgres от Supabase.)

## Меню-иерархия ≠ структура данных

Две независимые вещи, их легко спутать (отсюда «всё одним списком», что и раздражало):

1. **Группировка пунктов меню** (навигация слева) — коллекции складываются в **папки-группы**. Это организационный слой админки, папки не хранят данные. Делаем: «Каталог», «Материалы/услуги», «Ценообразование», «SEO/маркетинг», «Заказы».
2. **Иерархия внутри данных** — связи в модели:
   - **Категории как отдельная коллекция** со ссылкой материала на категорию (один уровень — достаточно для большинства).
   - **Самоссылающееся дерево** — поле `parent` на саму коллекцию (для деревьев любой глубины: Материалы → Бумага → Мелованная → 300 г).

> **Важно про ожидания:** группировка в папки и древовидное отображение — это **настройка при проектировании**, а не поведение по умолчанию. Из коробки Directus покажет плоский список. «Куча», которую видели раньше, — это непройденный этап настройки, не ограничение инструмента.

## Анти-паттерн модели

❌ **Не** сваливать разнородное в одну универсальную коллекцию «Сущности» с полем «тип» и разгребать фильтрами. У бумаги и у доставки нет общих полей — форма редактирования станет простынёй из полупустых ячеек. ✅ **Отдельная коллекция под каждую осмысленно разную сущность**, связи — ссылками. Тогда в каждой форме только релевантные поля.

## Коллекции (целевая модель)

Числа/значения — условные; вносятся в админке.

### Папка «Каталог»
- **`products`** — `name`, `slug`, **`тип_расчёта`** (`computed`/`fixed`), `категория` (→ `categories`), `опции` (⇉ `finishing_options`).
- **`categories`** — `name`, `parent` (↻ self, дерево).

### Папка «Материалы / услуги»
- **`papers`** — `name`, `плотность`, `ступени_цены` (⇉ `price_tiers`).
- **`finishing_options`** — `name`, `тип` (`per-unit`/`fixed`), `цена`. (Применимость к продуктам — через связь с `products`.)

### Папка «Ценообразование»
- **`price_tiers`** — `бумага` (→ `papers`), **`от_тиража`** (только нижняя граница!), `цена`, **`тип`** (`sheet`/`batch`). См. doc 03.
- **`fixed_prices`** — `продукт` (→ `products`), `тираж`, `цена`. Для продуктов с `тип_расчёта = fixed`.

### Папка «SEO / маркетинг» — продвигаемые страницы (pSEO)
- **`promoted_pages`** — см. отдельный раздел ниже.
- **`faq_items`** — вопросы-ответы (связываются со страницами услуг и promoted_pages).

### Папка «Заказы»
- **`orders`** и связанные — оформление, статусы, загруженные макеты, результат preflight, согласование цветопробы. (Детализация — отдельно при реализации.)

Обозначения: `→` many-to-one · `⇉` список (one/many-to-many) · `↻` self (дерево).

## Ввод тиражных ступеней в админке (UX данных)

Связь «бумага → много ступеней» отображается как **вложенный редактируемый список прямо в карточке бумаги** (штатный паттерн Directus для one-to-many). Открыл «Мелованная 300 г» — под ней список строк-ступеней с кнопкой «+».

- Каждая строка: `от_тиража`, `цена`, `тип` (`sheet`/`batch`).
- Сортировать по `от_тиража` по возрастанию — читается как диапазоны сверху вниз.
- **Почему это лучше JSON-файла** (который был инстинктивным выбором): та же структура, но с валидацией (цена — число, граница — целое, не пусто), формами без знания синтаксиса, сортировкой и **откатом ошибочной правки** (ревизии).
- Автоподпись диапазона («действует 20–39 шт») из коробки не появится — читается по соседней строке или настраивается кастомным отображением поля.

## `promoted_pages` — сущность продвигаемых комбинаций (pSEO)

Центральная для программного SEO (логика отбора — doc 05). Поля:

- `slug` (URL) · `h1`/`title`
- **`продукт`** (→ `products`) — какой конфигуратор открыть
- **`предустановка_опций`** (⇉ `finishing_options`) — что предвыбрать в конфигураторе
- `meta_title`, `meta_description`
- `intro_text` (rich text)
- `галерея` (⇉ `files`)
- `faq` (⇉ `faq_items`)
- **`статус`** (`draft`/`published`)

**Разделение «тянет / хранит»:**
- **Тянет из каталога:** продукт (конфигуратор) + предустановку опций.
- **Хранит своё:** URL, мета, заголовок, текст, галерею, FAQ — то, чего нет на родительской странице. Если заполнить нечем — страница не должна существовать (фильтр контента, doc 05).

**Генерация страниц:**
```
promoted_pages (статус = published)
  → Astro getStaticPaths по списку записей
  → N статических страниц (свой URL, мета, контент, предустановка конфигуратора)
```

**Предохранитель против спама (П5):** генерация **только** из записей `published`. Страница появляется, лишь когда человек осознанно завёл запись и наполнил её. **Никакого декартова перебора опций.**

**Технические требования к каждой pSEO-странице (см. doc 05):**
- свой `canonical` на саму себя (НЕ на родительскую — иначе самодедупликация);
- состояние конфигуратора — из URL/параметров (расшаривание + осмысленный адрес для Google);
- внутренние ссылки: родитель ↔ под-вариант (помогает Google понять иерархию, снижает каннибализацию).

## Где сидит движок относительно Directus

Движок цен (doc 03) — в коде (серверные функции Astro или отдельный сервис). Он **читает** ставки/бумаги/ступени/fixed-прайс из Directus через API и считает. В Directus — только справочные значения и контент (П1).
