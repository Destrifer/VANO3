# PLAYBOOK · Сбор семантики → архитектура раздела (повторяемый процесс)

**Это входная точка.** Триггер: *«давай соберём аналогичное по <продукту>»* (наклейки, листовки, буклеты…) → читай этот файл и иди по шагам. Продукт-агностично; пример реализации — раздел «визитки».

Связанные доки: стратегия — `../05-seo-content.md`; тех-реализация — `../06-seo-implementation.md`; пайплайн данных — `README.md`. Результаты по визиткам — `keyword-tree.md`, `site-architecture-vizitki.md`.

## 1. Стратегия (зачем так)

- **Хаб-и-кластер, интент-ориентированно:** один поисковый интент = одна страница ИЛИ вхождение. Не страница-на-ключ, не страница-на-продукт (доки 05 §7).
- **Хаб** головного интента (печать <продукта>) + **плитки** на нём → посадочные модификаторов + **хвостовые страницы** в sitemap + **вхождения** в текст.
- **Глубина > ширины:** меньше страниц, но каждая со спросом и своей механикой. Анти-паттерн — сотни тонких/дорвеев (05 §36).
- **Дифференциация механикой, а не заголовками:** каждая посадочная даёт что-то уникальное в UX (preview, мгновенная цена, точный срок, шаблон, preflight), а не «то же с другим H1».
- **Их пробелы = наша фора:** конкуренты часто без JSON-LD/FAQ и без части материалов — туда и бьём.

## 2. Уровни страниц (taxonomy решений)

| Уровень | Что | Критерий |
|---|---|---|
| **ХАБ** | 1 страница головного интента | всегда |
| **Tier-1 (плитки)** | посадочные модификаторов, видимые ссылки на хабе | спрос заметный + можем делать + отдельная потребность + есть уникальный контент/механика |
| **Tier-2 (хвост)** | отдельные страницы в sitemap, без плитки | НЧ/микро-НЧ + есть уникальный контент; иначе → Tier-3. Механизм: поле `promoted_pages.show_as_tile=false` (см. 06) |
| **Tier-3 (вхождение)** | только текст внутри страниц | синонимы головы, цена/тираж/гео, параметры конфигуратора (каннибализация) |
| **Гайд** | блог/инфо | информационный интент (как сделать, размер, шаблон) |
| **Исключить** | — | вне возможностей (capability) или чужой бизнес (nontarget) |

## 3. Повторяемый процесс (шаги)

### Шаг 1 — Сбор (делает владелец)
- Инструмент: **Я.Вебмастер → «Подбор запросов и анализ рынка (β)»** (по подтверждённому сайту). Лучше Вордстата: даёт спрос + клики + конкурентность + кластер.
- **Регион — Москва** (обязательно, иначе РФ).
- **Сиды:** широкие дают основной хвост, узкие почти дублируют → бери прежде всего:
  `<продукт>`, `печать <продукта>`, `изготовление <продукта>`, `заказ <продукта>` (+ 1–2 крупных модификатора при сомнении).
- **Минус-слова:** базовый список — `minus-words.txt` (Уровень 1+2) **плюс продукт-специфичные** (capability-фильтр, см. Шаг 3).
- Экспорт **XLS** → положить в `seo/queries/` или `Downloads/`. Имя файла = сид.

### Шаг 2 — Обработка (делает Claude)
```
python _pipeline.py <product>_master <file1.xlsx> <file2.xlsx> ...
```
Парсит xlsx (без внешних либ), мёржит, классифицирует, кластеризует, считает конкурентов → `report_clusters_<tag>.txt`, `report_competitors_<tag>.txt`, `data/queries_<tag>.csv`, `data/competitors_<tag>.csv`.

### Шаг 3 — Чистка и capability-фильтр
Классификатор делит на **PRINT / INFO / BRANDED / NONTARGET** (стоп-листы в `_pipeline.py`).
- **Capability-фильтр (важно!):** то, что мы НЕ делаем, — нецелевое, даже при спросе. Для визиток: офсет (нет оборудования), дизайн (нет ресурса) → ограниченно. Для нового продукта определить аналогично и добавить в минус-слова/NONTARGET.
- BRANDED — навигация на конкурентов, не цель (но сигнал рынка).

### Шаг 4 — Дерево интентов (`keyword-tree-<product>.md`)
- **Схлопнуть** дробные Яндекс-кластеры в смысловые ветки (синонимы в один интент — 05 §79).
- Верхний уровень коммерческих веток → ранжир по спросу → решение (хаб/landing/intext/guide/reject).
- **Проверка насыщения:** считать % эксклюзива по сидам. Узкие модификатор-сиды дают <15% нового → база исчерпана, дробить дальше не нужно.

### Шаг 4.5 — Tier-разметка всего ядра + ручная модерация (QA распределения)

Перед вёрсткой архитектуры — проставить **Tier каждому запросу** ядра и посмотреть распределение. Две цели: (а) убедиться, что страницы оправданы спросом и ни одна плитка не «пустая»; (б) поймать запросы, что **проскочили фильтр** (capability/nontarget) — это и есть ручная модерация ядра.

**Алгоритм — правила по приоритету, первое совпадение побеждает:**
1. **МОДЕРАЦИЯ / EXCLUDE (проверяем первым!):** мем/нерелевант · инфо · «не умеем» по capability (для визиток — офсет) · электронная/NFC/видео · DIY/шаблон/генератор · гео не наш регион · не-печать (аксессуар/носитель, напр. флешка-визитка). → вычеркнуть из ядра.
2. **Tier-1:** запрос ложится на действующую/планируемую плитку (своя ось модификации: техника/материал/форма/тир).
3. **Tier-2 (техн.):** отдельная техника/материал с НЧ-спросом, но своим контентом (визитки → ламинация, УФ-лак, шелкография).
4. **Ниши (Tier-2 потенц.):** аудиторные (профессия/отрасль) — страница только при своём контенте (галерея/шаблон под нишу); иначе Tier-3.
5. **Tier-3 (вхождение):** голова-синонимы · цена/недорого · тираж · гео-наш · доставка · формат/размер · ориентация · двусторонние · дизайн/макет.
6. **Fallback** → пометить «проверить» (при хорошем покрытии не должно оставаться).

**Выход:** распределение (кол-во запросов + сумма спроса по Tier) · список EXCLUDE · флаги на ручную проверку. Результат хранить рядом с ядром: `…_Tier.xlsx` (колонки `Запрос·Спрос·Tier·Целевая страница·Флаг`).

**Sanity-baseline (визитки, 485 запросов) — ориентир «здоровой» картины:**

| Tier | Доля спроса |
|---|---|
| Tier-3 (голова → хаб) | ~84% |
| Tier-1 (плитки) | ~12% |
| Tier-2 техн. + ниши | ~2% |
| EXCLUDE (модерация) | ~1,5% |

Голова **доминирует и живёт на хабе**, плитки берут различимые оси, Tier-2 тонкий. Если у нового продукта Tier-1 забирает сильно больше/меньше — проверить дробление (слишком крупно/мелко). Заметный EXCLUDE (визитки: офсет 523, флешка-визитка ~956) — норма, ядро всегда чистится вручную.

### Шаг 5 — Архитектура (`site-architecture-<product>.md`)
- ХАБ + таблица **вхождений** (синонимы головы, цена, тираж, гео, доставка, параметры).
- **Tier-1 плитки** с «от X ₽» + колонка **«чем уникальны (механика/UX)»** на наших активах (конфигуратор, `Preview.vue`, preflight, точный срок, шаблон, свотчи).
- **Tier-2** хвост, **Tier-3** вхождения, исключения.
- **Сверка с конкурентами:** реальная лига vs веб/генераторы; их плитки/под-деревья; их пробелы и фишки (галерея-табы, отзывы-виджет, ценовая лесенка).

### Шаг 6 — Реализация
По `../06-seo-implementation.md`: тех-фундамент (Этап 1) → хаб (Этап 3) → посадочные/pSEO (Этап 4). Контент — в Directus; тексты «ИИ оформляет факты владельца» (05 §88).

## 4. Зафиксированные правила-решения (применять всегда)

- Регион **Москва**; гео-модификаторы (метро/районы) — **НЕ страницы** (наш регион; анти-дорвей, 05 §37).
- Цена/недорого, тираж (100/1000/опт), двусторонние/односторонние — **вхождения**, не страницы (каннибализация).
- Синонимы головы (изготовление/заказ/производство/типография <продукта>) — **вхождения в хаб**.
- Узкие модификаторы дробить отдельными сидами **не нужно** (диминишинг — проверено).
- Дифференциация — **механикой**, не заголовками.
- Каждой посадочной — **уникальный контент** (своя галерея/специфика/FAQ), иначе это вхождение.

## 6. Чеклист «аналогичное по новому продукту»

Пример: **наклейки**.
1. Определить **capability-фильтр**: что НЕ делаем (напр. наклейки на стену/обои/тату/маникюр — если не наше) → минус-слова.
2. Сиды (Москва, минус ур.1+2 + продуктовые): `наклейки`, `печать наклеек`, `изготовление наклеек`, `заказ наклеек`, `стикеры`.
3. XLS → `python _pipeline.py naklejki_master <files>`.
4. Claude строит `keyword-tree-naklejki.md`, **Tier-разметку ядра `naklejki_Tier.xlsx` (Шаг 4.5, с модерацией)** и `site-architecture-naklejki.md` по шагам 3–5.
5. Сверка с конкурентами, дифференциация, число страниц.
6. Реализация по `06-seo-implementation.md`.

**Образец 1 — визитки (факт на 2026-06-15):** хаб `/business-cards` + **7 плиток Tier-1** (`urgent, foil, plastic, designer, magnetic, metal, round`) + **VIP** (Tier-1). **Tier-2** (`show_as_tile=false`): ламинация, УФ-лак. Ниши (психолог/массажист/такси/…) — резерв Tier-2 под контент. Эталон масштаба «глубоко, не широко».

**Образец 2 — наклейки (факт на 2026-06-15):** ТЗ — `site-architecture-naklejki.md`. Хаб `/stickers` + **8 плиток Tier-1** + **10 Tier-2** + **3D на `strategy=fixed`** (`/volume-stickers`). Отдельные хабы (этикетки/упаковка/штрихкоды) — STANDALONE. Особенности этого кейса:
- **Миграция «продукты → кластеры»:** модификаторы были 24 отдельными продуктами → перевели в `promoted_pages` под `stickers`, старые сняли с публикации (`status=draft`). **Редиректы не нужны, если сайт ещё не в индексе.**
- **`strategy=fixed`** — для технологий с иным ценообразованием (3D/смола): `fixed_price` (₽/лист по числу листов) + `fixed_sheet_*`, без бумаг/резки (всё включено). Первый такой продукт в проекте.
- **Новые материалы под кластеры** заводятся в `papers` + junction `products_papers`; пресет кластера = `paperIndex` в порядке материалов продукта.
- Цены новых материалов/3D — **черновые** (по конкурентам), помечать на выверку владельцу.

**Образец 3 — брошюры/буклеты/каталоги/меню/этикетки (факт на 2026-06-15):** раскатали модель на остальные крупные разделы. Уроки:
- **Три типа калькулятора → три ветки пресета** (`calcUrlState.ts`): *sheet* (`applyPreset`: `shape/sides/sizeIndex/foldTypeIndex/paperIndex/foil/laminationIndex/finishing[]`), *multipage* (`applyMultipagePreset`: `formatIndex/bindingId/pages`), *fixed* (своего пресета нет). Тип кластерного хаба определяет, какие поля пресета доступны.
- **«Листовка = буклет с 0 сложений»:** добавили фальц `folds:0` + гард в движке (биговку не считаем при folds=0); листовка/флаер = кластеры буклета с разными размерами (анти‑каннибализация). Аналогично «каталог = многостраничник» (catalogs перенастроили sheet→multipage).
- **Продукт мог быть пустым/не того типа** (catalogs/menus/labels были пустыми sheet без размеров) — при сборке донастраиваем продукт (размеры/переплёты/финиши/бумаги) ПЕРЕД контентом, иначе конфигуратор не считает.
- **Свой хаб vs вложение:** крупный раздел с самостоятельным интентом (этикетки 159k, B2B‑маркировка) — **отдельный хаб + кросс‑ссылки**, не кластер соседнего (даже если продукт похож). Вкладывать — только настоящий под‑режим (листовка в буклет).
- **Capability‑EXCLUDE по словам владельца:** офсет, широкоформат/баннеры, рулонные/флексо‑этикетки, авто/DTF/голография — в EXCLUDE, даже при спросе.

> ⚠️ **Gotcha (Vue острова):** новые `<select>` в конфигураторе — **только `v-model`** (через computed‑прокси при emit‑интерфейсе). Биндинг `:value+@change` теряет значение при гидрации острова → серверный пресет (`paperIndex`) сбрасывается. Проверять пресет **в превью после гидрации**, а не только в SSR‑HTML.
