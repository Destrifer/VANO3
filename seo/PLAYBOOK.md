# PLAYBOOK · Сбор семантики → архитектура раздела (повторяемый процесс)

**Это входная точка.** Триггер: *«давай соберём аналогичное по <продукту>»* (наклейки, листовки, буклеты…) → читай этот файл и иди по шагам. Продукт-агностично; пример реализации — раздел «визитки».

Связанные доки: стратегия — `../05-seo-content.md`; тех-реализация — `../06-seo-implementation.md`; пайплайн данных — `README.md`. Результаты по визиткам — `keyword-tree.md`, `site-architecture-vizitki.md`.

---

## 1. Стратегия (зачем так)

- **Хаб-и-кластер, интент-ориентированно:** один поисковый интент = одна страница ИЛИ вхождение. Не страница-на-ключ, не страница-на-продукт (доки 05 §7).
- **Хаб** головного интента (печать <продукта>) + **плитки** на нём → посадочные модификаторов + **хвостовые страницы** в sitemap + **вхождения** в текст.
- **Глубина > ширины:** меньше страниц, но каждая со спросом и своей механикой. Анти-паттерн — сотни тонких/дорвеев (05 §36).
- **Дифференциация механикой, а не заголовками:** каждая посадочная даёт что-то уникальное в UX (preview, мгновенная цена, точный срок, шаблон, preflight), а не «то же с другим H1».
- **Их пробелы = наша фора:** конкуренты часто без JSON-LD/FAQ и без части материалов — туда и бьём.

## 2. Уровни страниц (taxonomy решений)

| Уровень | Что | Критерий |
|---|---|---|
| **ХАБ** | 1 страница головного интента | всегда |
| **Tier-1 (плитки)** | посадочные модификаторов, видимые ссылки на хабе | спрос заметный + можем делать + отдельная потребность + есть уникальный контент/механика |
| **Tier-2 (хвост)** | отдельные страницы в sitemap, без плитки | НЧ/микро-НЧ + есть уникальный контент; иначе → Tier-3 |
| **Tier-3 (вхождение)** | только текст внутри страниц | синонимы головы, цена/тираж/гео, параметры конфигуратора (каннибализация) |
| **Гайд** | блог/инфо | информационный интент (как сделать, размер, шаблон) |
| **Исключить** | — | вне возможностей (capability) или чужой бизнес (nontarget) |

## 3. Повторяемый процесс (шаги)

### Шаг 1 — Сбор (делает владелец)
- Инструмент: **Я.Вебмастер → «Подбор запросов и анализ рынка (β)»** (по подтверждённому сайту). Лучше Вордстата: даёт спрос + клики + конкурентность + кластер.
- **Регион — Москва** (обязательно, иначе РФ).
- **Сиды:** широкие дают основной хвост, узкие почти дублируют → бери прежде всего:
  `<продукт>`, `печать <продукта>`, `изготовление <продукта>`, `заказ <продукта>` (+ 1–2 крупных модификатора при сомнении).
- **Минус-слова:** базовый список — `minus-words.txt` (Уровень 1+2) **плюс продукт-специфичные** (capability-фильтр, см. Шаг 3).
- Экспорт **XLS** → положить в `seo/queries/` или `Downloads/`. Имя файла = сид.

### Шаг 2 — Обработка (делает Claude)
```
python _pipeline.py <product>_master <file1.xlsx> <file2.xlsx> ...
```
Парсит xlsx (без внешних либ), мёржит, классифицирует, кластеризует, считает конкурентов → `report_clusters_<tag>.txt`, `report_competitors_<tag>.txt`, `data/queries_<tag>.csv`, `data/competitors_<tag>.csv`.

### Шаг 3 — Чистка и capability-фильтр
Классификатор делит на **PRINT / INFO / BRANDED / NONTARGET** (стоп-листы в `_pipeline.py`).
- **Capability-фильтр (важно!):** то, что мы НЕ делаем, — нецелевое, даже при спросе. Для визиток: офсет (нет оборудования), дизайн (нет ресурса) → ограниченно. Для нового продукта определить аналогично и добавить в минус-слова/NONTARGET.
- BRANDED — навигация на конкурентов, не цель (но сигнал рынка).

### Шаг 4 — Дерево интентов (`keyword-tree-<product>.md`)
- **Схлопнуть** дробные Яндекс-кластеры в смысловые ветки (синонимы в один интент — 05 §79).
- Верхний уровень коммерческих веток → ранжир по спросу → решение (хаб/landing/intext/guide/reject).
- **Проверка насыщения:** считать % эксклюзива по сидам. Узкие модификатор-сиды дают <15% нового → база исчерпана, дробить дальше не нужно.

### Шаг 5 — Архитектура (`site-architecture-<product>.md`)
- ХАБ + таблица **вхождений** (синонимы головы, цена, тираж, гео, доставка, параметры).
- **Tier-1 плитки** с «от X ₽» + колонка **«чем уникальны (механика/UX)»** на наших активах (конфигуратор, `Preview.vue`, preflight, точный срок, шаблон, свотчи).
- **Tier-2** хвост, **Tier-3** вхождения, исключения.
- **Сверка с конкурентами:** реальная лига vs веб/генераторы; их плитки/под-деревья; их пробелы и фишки (галерея-табы, отзывы-виджет, ценовая лесенка).

### Шаг 6 — Реализация
По `../06-seo-implementation.md`: тех-фундамент (Этап 1) → хаб (Этап 3) → посадочные/pSEO (Этап 4). Контент — в Directus; тексты «ИИ оформляет факты владельца» (05 §88).

## 4. Зафиксированные правила-решения (применять всегда)

- Регион **Москва**; гео-модификаторы (метро/районы) — **НЕ страницы** (наш регион; анти-дорвей, 05 §37).
- Цена/недорого, тираж (100/1000/опт), двусторонние/односторонние — **вхождения**, не страницы (каннибализация).
- Синонимы головы (изготовление/заказ/производство/типография <продукта>) — **вхождения в хаб**.
- Узкие модификаторы дробить отдельными сидами **не нужно** (диминишинг — проверено).
- Дифференциация — **механикой**, не заголовками.
- Каждой посадочной — **уникальный контент** (своя галерея/специфика/FAQ), иначе это вхождение.

## 6. Чеклист «аналогичное по новому продукту»

Пример: **наклейки**.
1. Определить **capability-фильтр**: что НЕ делаем (напр. наклейки на стену/обои/тату/маникюр — если не наше) → минус-слова.
2. Сиды (Москва, минус ур.1+2 + продуктовые): `наклейки`, `печать наклеек`, `изготовление наклеек`, `заказ наклеек`, `стикеры`.
3. XLS → `python _pipeline.py naklejki_master <files>`.
4. Claude строит `keyword-tree-naklejki.md` + `site-architecture-naklejki.md` по шагам 3–5.
5. Сверка с конкурентами, дифференциация, число страниц.
6. Реализация по `06-seo-implementation.md`.

**Результат по визиткам для образца:** 9 базовых страниц (хаб + 8 плиток), до ~13 с Tier-2. Это эталон масштаба «глубоко, не широко».
