# 03 · Ценовой движок

Самый «кодовый» документ. Движок живёт в коде (П1), читает справочные данные из Directus (doc 04), и является единственным источником истины по цене (П2).

> **Текущая реализация (этап «цифра»).** Движок (`src/lib/pricing/engine.ts`) считает только цифровую печать: тиражные ступени за лист (отдельно `4+0` и `4+4`), плоская цена бумаги за лист, постпечать (`setup` + за единицу; фольга — ступенчато), **резка** (плоттер/ручная), порог минимального заказа и округление. Данные маппятся из Directus в `data.ts`; зависимости полей — в `rules.ts`. Автовыбор «цифра ↔ офсет», множитель срочности (убран из UI — решает менеджер) и стратегия `fixed` заложены в дизайне ниже, но ещё не реализованы.

## Две стратегии расчёта — одна абстракция

У каждого продукта есть поле **`тип_расчёта`**:

- **`computed`** (параметрический) — сложный движок: лист, печать, постпечать, тиражные пороги, цифра-против-офсета.
- **`fixed`** (табличный / lookup) — никакой математики, справочник «тираж → конечная цена». Движок ищет строку, а не вычисляет.

**Критично (анти-паттерн — два калькулятора):** это НЕ два независимых калькулятора. Это **один интерфейс** «дай цену для конфигурации», за которым две реализации. Общим остаётся весь поток (выбор → цена → загрузка → заказ); различается только функция получения цены. Иначе придётся дублировать загрузку, preflight, оплату, корзину.

**Стратегий должно быть мало и они по-настоящему разной природы.** `computed` vs `fixed` — разные природы (вычисление vs поиск). А «computed с другой формулой для баннеров» — это НЕ новая стратегия, это параметры внутри `computed`. Разнообразие продуктов поглощается данными, не кодом.

**Когда продукт делают `fixed`:** цена пришла от подрядчика готовой (перепродажа); себестоимость не раскладывается на компоненты; акционный прайс «как на рынке». Суть: цена — это входные данные, а не результат вычисления.

## Стратегия `computed` — порядок операций

### Три типа ценового воздействия (не смешивать)

1. **Поштучно (per-unit)** — масштабируется с тиражом: печать, бумага, ламинация.
2. **Фиксированно (fixed)** — не зависит от тиража: подготовка файла, форма/пластина для офсета, приладка, настройка постпечати. **Именно фиксированные делают малый тираж дорогим за штуку.**
3. **Множитель (multiplier)** — процент ко всему: двусторонняя (×~1.55), срочность (×~1.5). Не плюс, а коэффициент.

### Порядок (строго; порядок и есть суть)

```
1. Печать за единицу = базовая_ставка_технологии × коэф_формата × множитель_сторон
2. + бумага (поштучно)
3. + постпечать поштучная
   = СЕБЕСТОИМОСТЬ ЕДИНИЦЫ
4. × тираж  (с учётом тиражных ступеней — см. ниже)
   = подытог печати
5. + фиксированные (подготовка + приладка/настройка технологии + фикс. постпечати)
   = подытог
6. × глобальные множители (срочность) — в самом конце
7. порог минимального заказа (floor)
8. округление (до разумных денег, напр. вверх до €0.50)
```

> Главная ошибка — всё просто сложить. Порядок объясняет экономику печати (почему 100 шт дорогие за штуку, а 1000 — дешёвые: фиксированные «размазываются»).

### Выбор технологии (цифра vs офсет) — автоматический

- Цифра: низкие фиксированные (setup), высокая ставка за единицу.
- Офсет: высокая приладка (пластины), низкая ставка за единицу.
- **Движок считает оба варианта и берёт дешёвый.** Не привязывать технологию к продукту намертво. Точка перехода зависит от тиража и постпечати.

## Тиражные ступени (для `computed`) — диапазонная сетка

Цена за лист падает с тиражом **ступенчато** (bracket pricing), не линейно. Хранится как отдельная коллекция (doc 04).

### Модель «только нижняя граница» (ОБЯЗАТЕЛЬНО)

Каждая ступень хранит **только нижнюю границу** `от_тиража`, ставку и тип. Верхняя граница выводится из следующей ступени.

- ❌ Модель «от и до» (две границы) — источник багов: дыры, нахлёсты, ручная правка соседей.
- ✅ Модель «только от»: «от 10 → ставка1», «от 20 → ставка2», «от 40 → ставка3». Диапазоны (10–19, 20–39, 40+) выводятся автоматически. Дыр и нахлёстов физически не может быть. Как налоговые скобки.

> **Ниже нижней ступени (важно).** Минимума по тиражу/числу листов нет, поэтому заказ может дать число листов меньше нижней ступени. В этом случае `tierRate()` берёт ставку **самой нижней ступени** (самую дорогую за лист), а НЕ 0 — иначе печать/ступенчатая постпечать считались бы бесплатно. «Пол» рентабельности при этом обеспечивает `minOrder` на итоге, а не нулевая ставка. Поэтому нижняя ступень обязана нести реальную цену за лист.

### Тип ставки в ступени (КРИТИЧНО)

В одной колонке «цена» могут быть РАЗНЫЕ по смыслу числа. Обязательно поле **`тип`**:
- **`sheet` (за лист)** → ставка × тираж.
- **`batch` (за тираж)** → ставка как есть (фиксированная за весь диапазон).

Без этого поля движок посчитает чушь (30 «за тираж» и 25 «за лист» выглядят одинаково, означают разное).

### Режим применения ступени

- **Норма для полиграфии:** ставка достигнутой ступени применяется **ко всему тиражу** (не прогрессивно). Попал в ступень — вся партия по её цене.
- (Альтернатива — прогрессивно, как налог: первые N по старой ставке, остаток по новой. Для печати почти не используется.)
- **Зафиксировать режим в движке явно** — по одним данным два режима дают разные числа.

### Известный артефакт (фича, не баг)

При режиме «вся партия по новой ставке» переход в более дешёвую ступень может сделать **больший тираж дешевле меньшего** (напр. 40 шт дешевле 39). Это:
- нормальное поведение модели — понимать, не пугаться;
- основа подсказки клиенту «добавьте 1 шт до 40 и заплатите меньше» (приятно + растит средний чек).
- если в каком-то продукте нежелательно — лечить **данными** (подобрать ставки), не переписыванием движка.

## Стратегия `fixed` — табличный расчёт

Простой справочник «тираж → конечная цена». Без ставок за лист, без графика-наклона.

**Промежуточные тиражи (бизнес-решение, реализуется в коде):**
- ✅ Рекомендация для старта: **только значения из таблицы** (выпадающий список, не свободный ввод) — исключает двусмысленность, отражает суть «у нас на это просто есть прайс».
- Альтернативы: округление вверх до ближайшего тиража; интерполяция между строками. Усложнять позже при необходимости.

Логика выбора строки — в коде; числа — в данных (П1).

## Резка (считается автоматически, пользователь не выбирает)

Резка — скрытая статья себестоимости; ветка зависит от того, на чём режут (та же логика, что выбор листа — `isPlotter`):

- **Плоттер** (плоттерный лист или непрямоугольная/круглая форма) — ступенчатый тариф по числу листов. Диапазоны заданы **верхней границей** `to` (до N листов → фикс. цена за весь заказ). Это зеркальная к тиражным ступеням модель: там «только нижняя граница», здесь — «только верхняя», но принцип тот же — один поиск по таблице, без ручных вилок «от-до». Реализация: `bracketRate()`.
- **Большой лист** (ручная резка) — процент от заказа (сейчас `0.15` = 15% от суммы печать + бумага + постпечать).

Данные — в `pricing_settings`: `plotter_cutting` (JSON `[{to, price}]`) и `manual_cutting_rate`. Числа — в Directus, логика — в коде (П1).

## Предохранители (обязательны)

- **Единый источник истины (П2)** — витрина и оформление считают одним движком.
- **Порог минимального заказа / рентабельности** — никакая комбинация не опускает цену ниже себестоимости. У движка есть «пол».
- **Грациозный отказ** — если комбинация за пределами честного расчёта (экзотика, гигантский тираж) — не угадывать, а передавать в B2B-трек на ручной расчёт (П3).
- **Округление** — единое правило (напр. вверх до €0.50), иначе вылезают цены вида €37.83.

## Связь с данными (doc 04)

- `computed` читает ставки технологий, коэффициенты форматов, цены бумаг, опции постпечати, **тиражные ступени** (`price_tiers`).
- `fixed` читает таблицу `fixed_prices`.
- Поле `тип_расчёта` продукта — переключатель и для движка (какую ветку считать), и для админки (какую ветку показать редактору).
- Вся логика (порядок операций, выбор технологии, поиск ступени, применение типа) — в коде. В админке только числа.
