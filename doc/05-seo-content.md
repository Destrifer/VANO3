# 05 · SEO и контент

## Страницы услуг — SEO-фундамент

У страницы услуги две работы: **ранжироваться** по одному запросу и **довести** холодного человека до цены. Конфигуратор-остров не индексируется (П6) — ранжируется статическая обвязка, и эта обвязка и есть страница услуги.

### Таксономия: одна страница = один поисковый интент

- НЕ одна страница на продукт, НЕ страница на ключевое слово — на **интент**.
- Модификатор заслуживает отдельной страницы, только если у него **своя устойчивая потребность и спрос** (напр. «срочная печать визиток» — другой человек, другая боль). «Визитки 350 г» — не интент, это опция в конфигураторе.
- Слишком крупно → не ловим длинный хвост (а новичок только по нему и ранжируется). Слишком мелко → почти одинаковые тонкие страницы конкурируют (каннибализация).

### Коммерческий vs информационный интент — разные страницы

- «Печать визиток» (заказать) → страница услуги с конфигуратором.
- «Как сделать макет визитки» (узнать) → гайд.
- Связываются ссылками, но не смешиваются. Информационный ответ как главный контент на коммерческой странице ухудшает обе.

### Анатомия страницы (сверху вниз)

1. **H1** дословно по запросу (не слоган).
2. **Первый экран:** что это, цена «от», вход в конфигуратор. Холодный посетитель за пару секунд понимает «мне сюда» (хороший клик / NavBoost).
3. **Конфигуратор** (остров).
4. **Характеристики** (текстом!): форматы, бумаги, постпечать, мин. тираж, сроки — **конкретными числами**. Главный SEO-актив + материал для AI-цитирования. Дублирует возможности конфигуратора текстом (для машины и для случая, когда остров не отрисовался).
5. **Примеры именно этого продукта** (не общий портфель). Реальные фото.
6. **Требования к макету + скачиваемый шаблон** (конфигуратор знает формат → задаёт шаблон) + ссылка на гайд.
7. **FAQ** (с разметкой FAQPage) — ловит длиннохвостые вопросительные запросы, идеален для AI-ответов.
8. **Внутренние ссылки** — вверх (категория), вбок (смежные услуги), на гайды.

### SERP-слой
`title` (≤60 знаков, содержит запрос + зацепку: срок / «от» цены / регион) и `meta_description` — то, что видно до клика, влияет на CTR.

### Тематическая сетка (авторитетность без закупки ссылок)
Хаб категории → страницы услуг → ссылки между собой по смыслу → гайды → обратно. Плотная сетка говорит Google «сайт про печать глубоко». Для нового домена — один из немногих способов растить авторитетность.

### Анти-паттерн (штрафуется с 2012)
❌ Сотни почти одинаковых страниц под все ключи, набивка ключевых слов, дорвеи под микрорайоны. ✅ Меньше страниц, но каждая глубокая (П4).

> Шаблон «что писать» по блокам — в файле `kontent-autlayn-stranica-uslugi.md` (отдельный артефакт).

## Программный SEO (pSEO)

Шаблон + данные = много страниц под длинный хвост, у каждой свой URL/мета/контент и **предустановка конфигуратора**. Astro генерирует их из источника данных на сборке (штатный сценарий). Сущность данных — `promoted_pages` (doc 04).

### Граница рабочего pSEO vs спама

Опасность НЕ в том, что страниц много автоматически, а в **переборе без пользы**. Фильтр П5 — **все три обязательны:**
1. **Спрос** — комбинацию реально ищут отдельно (подтверждение минимум из двух источников).
2. **Контент** — есть чем наполнить именно про неё (своя галерея, специфика, FAQ), чего нет на родителе.
3. **Намерение** — коммерческое (заказать), не информационное.

Прошёл все три → страница. Не прошёл → очередь ожидания, не генерация.

### Управление списком, не перебором
Генерировать только из явного списка `promoted_pages (published)`. Физически нельзя наплодить мусор. Объём найденных формулировок ≠ объём страниц.

### Технические требования pSEO-страницы
- свой `canonical` на себя (НЕ на родителя);
- родитель ↔ под-вариант ссылками (иерархия, анти-каннибализация);
- состояние конфигуратора в URL;
- уникальный контент, не подстановка одного слова. Не хватает контента на N страниц — значит страниц меньше N.

## Отбор запросов / комбинаций (процесс)

**Честная рамка:** точных объёмов сверхНЧ-запросов не существует ни у кого. Цель — не «найти запрос с объёмом X», а **набрать доказательства собственного интента триангуляцией**.

### Источники (по надёжности)
1. **Свой Google Search Console** — золото, но накопительный (на старте данных мало). После индексации показывает реальные формулировки, по которым уже показываемся. → Главный источник на втором этапе.
2. **Подсказки Google** — автодополнение, «Похожие запросы», «Люди также спрашивают». Бесплатные реальные запросы. Доступны сразу, до трафика.
3. **Вопросы клиентов/менеджеров** — интент в чистом виде до того, как стал поисковым. Записывать повторяющиеся. Кормят и FAQ, и отбор комбинаций.
4. **Платные инструменты ключей** — НЕ первыми: на длинном хвосте их цифры наименее надёжны (ловушка «показал 0 — значит не нужно»). Использовать для расширения списка формулировок и грубого ранжирования кандидатов, как порядок величины, не как факт.

### Три фильтра (как в pSEO) + группировка
- Прогнать каждого кандидата через **спрос + контент + намерение**.
- **Сгруппировать синонимичные формулировки в один интент ДО подсчёта страниц** («фольгирование» = «тиснение фольгой» = «золотой/серебряной фольгой» → одна страница, формулировки отрабатываются внутри в секциях/FAQ). Иначе сами устроите каннибализацию.

### Посевной запуск с проверкой
Не угадывать идеальные 20. Сделать качественно **3–5 очевидных** (всплыли и в подсказках, и в вопросах клиентов), опубликовать, через ~2 месяца открыть GSC: что зацепилось и по каким соседним формулировкам. Следующую волну отбирать на данных своего сайта (П7).

> Пример отбраковки (визитки): «фольгирование» — проходит (флагман); «дизайнерская бумага» — проходит (с учётом, что на неё фольга не ложится); «с QR-кодом» — отвал по контенту (нечего сказать сверх обычной печати); «двусторонние» — отвал по каннибализации (это параметр конфигуратора). Из десятка кандидатов до посева доходит 3–5 — это работающий фильтр, а не упущенные возможности.

## Генерация текстов с ИИ

### Позиция поисковика (факт, не миф)
Google **не наказывает за ИИ-авторство**. Наказывает за низкое качество и **scaled content abuse** (массовое производство ради позиций без пользы). Планка одна для ИИ и человека. → ИИ-тексты допустимы; защищает от scaled abuse наша pSEO-архитектура с отбором (П5), а не отказ от ИИ.

### Правильная конструкция: «ИИ оформляет ВАШИ факты», не «ИИ пишет из головы»
Слабое место ИИ — он не знает реального опыта производства (на какую бумагу фольга не ложится, ваш MOQ, какие цвета в наличии, сроки). Именно эти конкретные проверяемые факты отличают ранжируемую страницу от гладкой пустышки (это «experience» из E-E-A-T).

**Процесс:** владелец/менеджер даёт **факты** (короткими тезисами, не прозой) → ИИ оформляет в текст по шаблону (структура из контент-аутлайна: вводный абзац, особенности/ограничения, варианты, FAQ, призыв к конфигуратору) → **ручная модерация фактов** (не стиля).

### Два обязательных предохранителя
1. **Достоверность.** Модель может уверенно соврать про производство → клиент закажет по невыполнимому обещанию. Модерация фактов **обязательна**, не опция. При подходе «оформляет ваши факты» проверять легко (не переврал ли данное).
2. **Шаблонное однообразие.** Один промпт на 20 страниц → узнаваемо одинаковый ритм → читается как штамповка. Лекарство — **разнообразие входных фактов**, не правки. Если у двух комбинаций факты почти одинаковы — это сигнал, что и страниц должно быть не две, а одна (каннибализация).

### Итог
Важно не «кто написал», а «что в тексте». ИИ-оформление реальных фактов задачу выполняет; ИИ-генерация из воздуха — нет (накажут за пустоту, не за ИИ). Решает, поставите ли на вход правду о своём производстве.

> Шаблон-промпт для генерации (фактические слоты + ограничения против присочинения) — следующий артефакт к проработке, если понадобится.
