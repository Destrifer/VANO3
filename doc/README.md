# Типография — проектный бриф для разработки

Этот набор документов описывает сайт типографии (печатного салона) на **Astro** + **Directus**: что строим, как и — главное — какие границы нельзя нарушать. Документы написаны как контекст для ИИ-ассистента, помогающего писать код, и для самого владельца проекта.

## Как читать

Читайте по порядку. Первый документ задаёт контекст и сквозные принципы — без него остальные можно понять неправильно.

1. **[01-project-brief.md](01-project-brief.md)** — что за бизнес, какая архитектура в целом (два маршрута), сквозные принципы и анти-паттерны. **Начинать отсюда.**
2. **[02-frontend.md](02-frontend.md)** — Astro, островная архитектура, производительность, разметка, конфигуратор и preflight (UX-сторона), A/B-тесты, запреты.
3. **[03-pricing-engine.md](03-pricing-engine.md)** — ценовой движок: типы воздействия, порядок операций, две стратегии расчёта, тиражные ступени, предохранители. Самый «кодовый» документ.
4. **[04-backend-directus.md](04-backend-directus.md)** — модель данных, коллекции, меню-иерархия, ввод тиражных ступеней, сущность продвигаемых страниц.
5. **[05-seo-content.md](05-seo-content.md)** — страницы услуг, программный SEO (pSEO), отбор запросов, генерация текстов с ИИ.

## Главное в двух фразах

- Строим **онлайн-салон печати**: холодный трафик из поиска ведём через конфигуратор к заказу; B2B по сарафану — в трек с менеджером.
- Успех определяется не трюками, а **технической базой + честным конфигуратором + глубокими страницами под реальный спрос**, при жёстком разделении «данные в админке / логика в коде».

## Статус числовых данных

Все цены, ставки, пороги и тиражи в документах — **условные, для иллюстрации логики**. Реальные значения вносятся в админку владельцем; в код они не зашиваются (см. принцип разделения данных и движка).
