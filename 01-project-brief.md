# 01 · Проектный бриф и сквозные принципы

## Что за бизнес

Типография (печатный салон). Профиль работы: **заказы в основном через сайт + B2B по сарафанному радио**. Поток «с улицы» (walk-in) почти отсутствует. Из этого следует ключевой вывод, определяющий всю архитектуру:

**Сайт делает две разные работы для двух аудиторий, приходящих разными путями.**

| | Онлайн-заказчик | B2B-заказчик |
|---|---|---|
| Откуда | холодный, из поиска (SEO) | тёплый, по рекомендации |
| Что ищет | находит по запросу услуги | вбивает бренд / переходит по ссылке |
| Работа сайта | дать самому дойти до цены и заказа без менеджера | убедить, не отпугнуть, вывести на разговор |
| Что на это работает | SEO + конфигуратор | доверие + брендовый поиск + контакт |

Поэтому архитектура — **один сайт, два входа, две мерки успеха**. Не «средний сайт для всех».

## Стек

- **Фронт:** Astro (статика + острова интерактива + SSR-эндпоинты по необходимости).
- **Бэк / данные:** Directus (БД + автогенерируемая админка + REST/GraphQL API).
- **Ценовой движок:** код (серверные функции Astro или отдельный тонкий сервис), читает данные из Directus.

## Архитектура сайта (два маршрута)

**Главная** разводит на два входа.

**Маршрут 1 — Онлайн-заказ (главный мотор, воронка):**
Каталог услуг → Страницы услуг (SEO-посадочные) → Конфигуратор → цена `[остров]` → Загрузка макета + preflight `[остров]` → Оформление + оплата `[остров]` → Цветопроба/согласование `[остров]` → Доставка.

**Маршрут 2 — Для бизнеса (B2B):**
Страница «Для бизнеса» → Запрос расчёта `[остров: форма]` → Кейсы/проекты → Контакт с менеджером.

**Общие блоки (питают оба маршрута):** Портфолио · О нас/производство · Гайды по подготовке файлов + шаблоны · Гарантии/отзывы · Контакты/Google Business Profile.

`[остров]` = интерактивный компонент с JS (Astro island). Всё остальное — статический HTML.

> **GBP — гигиена, не мотор.** Поскольку walk-in мало, Google Business Profile нужен (брендовый поиск, AI-рекомендации, локальные B2B), но это не главный канал привлечения. Главный мотор роста — страницы услуг под онлайн-заказ.

## СКВОЗНЫЕ ПРИНЦИПЫ (нарушать нельзя)

Эти принципы важнее любой отдельной фичи. Если реализация конфликтует с ними — неправа реализация.

### П1. Данные — в админке, логика — в коде
Справочные значения (цены, бумаги, ставки, тиражные пороги, тексты страниц) живут в Directus и правятся без программиста. **Формулы и правила расчёта живут в коде.** Как только формула переезжает в данные/CMS — теряется тестируемость и версионирование, возвращается хрупкость. Граница: *что* (значения) — данные, *как* (применение) — код.

### П2. Цена из конфигуратора = цена в счёте, до копейки
Ценовой движок — единственный источник истины по цене. Витрина и оформление заказа считают одним и тем же кодом. Если менеджер потом «поправляет» автосчёт — доверие к онлайн-каналу рушится. Нельзя иметь два разных расчётчика.

### П3. Узкая зона честной цены лучше широкой и врущей
Конфигуратор считает мгновенно только стандартные, параметрические заказы. Нестандарт (экзотические материалы, спорная постпечать, переговорные тиражи) **не угадывается**, а уходит кнопкой «нужен расчёт» в B2B-трек. Лучше честно сказать «посчитаем индивидуально», чем показать неверную цифру.

### П4. Глубина важнее количества страниц
10 глубоких страниц под реальный спрос бьют 200 пустых. Это касается и страниц услуг, и программного SEO. Объём найденных запросов — не повод для объёма страниц.

### П5. Отбор по «спрос + контент + намерение», не по перебору
Любая авто-генерируемая страница появляется, только если у комбинации есть собственный поисковый интент, собственный уникальный контент и коммерческое намерение. Все три обязательны. Никаких декартовых произведений опций.

### П6. Краулеро-дружелюбный HTML
Контент, важный для SEO и для AI-краулеров (GPTBot, PerplexityBot и т.п.), должен присутствовать в исходном HTML без обязательного исполнения JS. Острова — только для интерактива, не для контента, который надо индексировать.

### П7. Петля обратной связи через Search Console
Любое SEO-решение (какие страницы делать, нет ли каннибализации) — это гипотеза до проверки в GSC. Закладываем GSC с первого дня; после публикации сверяемся: что зацепилось, не конкурируют ли страницы за один запрос.

## АНТИ-ПАТТЕРНЫ (так делать НЕ надо, даже если кажется удобным)

- ❌ Зашивать цены/правила в код или в интерфейс вместо данных в админке (нарушает П1).
- ❌ Делать два независимых калькулятора вместо одной абстракции с разными стратегиями (нарушает П2, см. doc 03).
- ❌ Пытаться посчитать в конфигураторе «вообще всё», включая экзотику (нарушает П3).
- ❌ Генерировать pSEO-страницы перебором всех опций (нарушает П5 — это дорвеи, штрафуется).
- ❌ Класть индексируемый контент только в JS-острова (нарушает П6).
- ❌ Делать бинарный preflight «годен/не годен» вместо трёх исходов (см. doc 02).
- ❌ Показывать Googlebot один контент, людям другой по user-agent (клоакинг, штрафуется; см. doc 02).
- ❌ Отдавать ИИ генерацию текста «из головы» без фактических вводных от владельца (см. doc 05).
- ❌ Сваливать разнородные сущности в одну универсальную коллекцию с полем «тип» (см. doc 04).

## Почему вообще так (краткая рациональная база)

Решения опираются на подтверждённые (документация Google + утечка Content Warehouse 2024 + показания на процессе DOJ) сигналы, а не на догадки SEO-блогов:

- **Клик-данные реальны** (система NavBoost: goodClicks / badClicks / длина последнего клика). → Отсюда упор на пост-клик опыт: страница должна закрывать запрос так, чтобы человек не вернулся в выдачу. Конфигуратор с мгновенной ценой — главный «хороший клик».
- **Core Web Vitals (вкл. INP) — подтверждённый сигнал.** → Отсюда упор на статику/острова и производительность.
- **Есть мера авторитетности сайта и испытательный период для новых доменов.** → Новый сайт не прыгнет в топ за 2–3 месяца; стратегия — безупречная база + длинный хвост + запуск циклов, которые компаундятся.
- **Google не наказывает за способ создания контента, наказывает за низкое качество и scaled content abuse.** → ИИ-тексты допустимы при отборе и реальных фактах на входе (doc 05).

Тон всех решений: скепсис к «магии», ставка на то, что в руках (техника, честность расчёта, реальный опыт), проверка гипотез данными.
