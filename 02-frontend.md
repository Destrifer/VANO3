# 02 · Фронтенд (Astro)

## Принцип рендеринга

Astro выбран потому, что у типографии контент в основном статический (каталог, страницы услуг, гайды, портфолио), а интерактив точечный. Целевая модель:

- **Статика / SSG** — каталог, страницы услуг, гайды, портфолио, страницы pSEO. Отдаётся готовым HTML мгновенно, отлично индексируется (П6).
- **Острова (client islands)** — только интерактив: конфигуратор, загрузка+preflight, оформление/оплата, цветопроба, B2B-форма. JS грузится только там.
- **SSR-эндпоинты / Astro actions** — серверный расчёт цены, обработка загрузки и preflight (см. ниже — это серверная работа).

Островная архитектура (частичная гидратация) даёт лучший INP при минимуме скриптов. Контент для индексации **не** прячем в острова (П6).

## Производительность (Core Web Vitals — подтверждённый сигнал)

- Закладывать производительность в архитектуру сразу, а не «оптимизировать потом».
- Целевые метрики: LCP, CLS, **INP** (ключевая метрика отзывчивости).
- Изображения (особенно портфолио и галереи) оптимизировать на сборке.
- **Speculation Rules API** — заложить для предсказуемых маршрутов (категория → товар, статья → следующая). Пререндер/предзагрузка вероятной следующей страницы → лучше TTFB/LCP, переходы почти мгновенные. Реализуется JSON-блоком на страницах. Большинство конкурентов этого не делает — дешёвое преимущество.

## Структурированные данные (Schema.org)

Закладывается дёшево, даёт rich-результаты и машинную читаемость для AI:

- **Service / Product + Offer** (цена «от», валюта) — на страницах услуг и pSEO.
- **FAQPage** — на блоках FAQ.
- **BreadcrumbList** — на хлебных крошках.
- **LocalBusiness / Organization** — глобально (для брендового и локального).

## Персонализация и клоакинг (ГРАНИЦА)

- **Персонализация разрешена**, если применяется единообразно ко всем в сегменте, включая краулера (который получает версию по умолчанию). Это инструмент **конверсии**, не ранжирования.
- **Клоакинг запрещён** (штрафуется): нельзя показывать Googlebot один контент, людям — другой, определяя бота по user-agent.
- Базовый индексируемый контент страницы должен быть **один и тот же для всех** и не подменяться целиком динамикой. Не использовать детекцию ботов.

## A/B-тестирование (если будем)

Прямой «интеграции с Google» не существует. Есть два рабочих механизма:

1. **SEO A/B-тест** — делим не пользователей, а страницы (часть карточек с шаблоном A, часть с B), смотрим разницу в показах/кликах/позиции в Search Console.
   - Серверный рендер вариантов (не клиентский).
   - Один элемент за раз.
   - Googlebot и пользователь видят по данному URL одно и то же.
   - Для тестов на отдельных URL — `canonical` варианта на оригинал + временный **302** (не 301), чтобы не потерять позиции оригинала.
2. **Search Console API** — источник правды о том, что зацепилось.

> Большинство CRO-тестов (цвет кнопки, текст CTA) не дают измеримого SEO-эффекта — бояться тестировать не нужно при соблюдении техники.

## Конфигуратор (остров) — UX и поток

Сердце онлайн-маршрута. Это «движок ценообразования с интерфейсом» (алгоритм — в doc 03). UX-требования:

**Поток — поступательный, не стена полей.** Раскрывать следующий шаг по мере выбора:
1. Продукт (часто уже выбран, если пришли со страницы услуги).
2. Ключевые параметры — формат/размер, бумага, красочность, одно-/двусторонняя.
3. Тираж — отдельный шаг (показывать разбивку «100 шт — X/шт, 500 — Y, 1000 — Z», мягко подталкивая к большему; уважать минимальный тираж MOQ).
4. Постпечать — **только применимые к этому продукту** опции (условная логика: фальцовка — у листовок, переплёт — у многостраничных; неприменимое скрывать, не делать серым).
5. Срок — стандарт / срочно.

**Сквозь все шаги — всегда видимая живая цена**, пересчёт мгновенный без перезагрузки. Ради этой реактивности конфигуратор и вынесен в остров. Это главный «хороший клик» (NavBoost).

**Показывать расчётную дату готовности** (конкретную дату, не «3–5 дней») — снимает тревогу у платящего вперёд незнакомцу.

**Граница зоны мгновенной цены (П3):** стандартные заказы считаются на лету; кнопка «Нужен расчёт» уводит нестандарт в B2B-трек.

**Выход конфигуратора — это спецификация.** После конфигурации система знает точный формат и нужные вылеты → предложить скачать шаблон ровно под этот заказ и задать правила для preflight. Конфигуратор → шаблон → preflight — **одна цепочка данных**.

**SEO-граница:** конфигуратор-остров не индексируется. Ранжируется статическая обвязка (страница услуги). Не пытаться делать SEO для калькулятора.

### Архитектура реализации (как разбит код)

Чтобы калькулятор не рос в монолит и переиспользовался между продуктами:

- **Логика цены — вне Vue:** `src/lib/pricing/engine.ts` (чистый конвейер расчёта), `data.ts` (Directus → типы движка), `rules.ts` (декларативные зависимости полей, напр. «фольга → ламинация Soft Touch»).
- **Состояние формы — композабл** `src/composables/useCalculator.ts`: всё реактивное состояние + сборка `OrderConfig` + `result`. Возврат обёрнут в `reactive()`, чтобы поля писали `calc.shape` напрямую.
- **Раздача состояния:** `ProductConfigurator.vue` — остров: создаёт состояние и отдаёт всем зонам (поля/превью/плашка) через `provide`/`inject` по типизированному `calcKey` (без проброса десятков пропсов).
- **Поля — презентационные компоненты** в `src/components/calculator/`: `SizeField`, `QuantityField`, `SidesField`, `MaterialField`, `FinishingField` (собраны в колонку `Calculator.vue`) + общий `SwatchPalette` (палитра цветов + lightbox).
- **Стили** — daisyUI/Tailwind по теме `lofi`; самописный CSS допускается только для раскладки (`grid-template-areas` в `ProductConfigurator`), без цветов/токенов.
- **Раскладка страницы** — равные колонки `grid-template-areas` (3 → 2 → 1): поля | превью+плашка | галерея. Мобайл: превью → поля → плашка → галерея. Галерея — `container-query` (2 миниатюры → 1).
- **Плашка заказа** — `OrderPlate.vue`: итог + «В корзину» (позиция для будущей корзины).

#### Превью (общий движок + макеты продуктов)

- **`src/lib/preview/primitives.ts`** — чистые ctx-хелперы (не зависят от продукта): `shapePath` (контур из формы/размера; позже — Path2D вырубки), `paperTexture`, `laminationGloss`, `drawFoilText` (металл), `inkColor`, цвет-утилиты.
- **`src/lib/preview/mockups.ts`** — реестр макетов: `content()` (печать) + `foil()` (металл). Реестр **пообъектный**: у каждого продукта свой образ, спроектированный от его кластеров. Принципы, прогресс и рецепт добавления сцены — **`07-preview-mockups.md`**.
- **`Preview.vue`** — общий stage (canvas/DPR/ResizeObserver/реактивность): контур → материя → макет → глянец → фольга. Макет выбирается по `product.previewKind` (по умолчанию `"card"`).
- Универсальные параметры (форма, размер, материал/цвет, ламинация, фольга, скругление) живут в `calc`; макет рисует **только ink-содержимое**. **Разные дефолты (круг/фольга) — не разные макеты**: различия кластеров = материал и форма, их даёт движок. Инвариант «кукла vs движок» и исключение из него — в `07-preview-mockups.md`.

Для будущих составных продуктов (брошюры: обложка + блок) поля переиспользуются, а движок расширяется на `parts[]`.

**Сохраняемость:** конфигурацию делать расшариваемой по ссылке (состояние в URL/параметрах, не только в JS) — для B2B («показать начальству») и как закладку. Это же помогает Google видеть осмысленный адрес.

## Загрузка макета + preflight (остров + сервер)

**Суть:** preflight = перенос приёмщика-печатника в код. Без живого приёмщика его работу делает либо preflight, либо печатная машина на испорченном тираже. Это защита маржи, не «приятная фича».

**preflight ≠ proofing (две разные проверки):**
- **Preflight** — техническая, машинная: годится ли файл физически для печати.
- **Proofing** — смысловая, человеческая: то ли это вообще, нет ли опечаток, тот ли логотип, тот ли цвет.
- Preflight **предшествует** цветопробе, не отменяет её.

**Проверка идёт против спецификации заказа** (из конфигуратора), а не «файла вообще»: ожидаемый размер с вылетами, число полос (1 или 2 для двусторонних), цветовая модель.

**Что проверяет (по убыванию частоты брака):**
1. Цветовая модель — CMYK vs RGB; нежелательные плашечные цвета.
2. Разрешение — < ~300 dpi в реальном размере = «мыло».
3. Размер и вылеты — совпадение формата; наличие вылета (2–3 мм); важный объект не в «опасной зоне» у реза.
4. Шрифты — зашиты/в кривых.
5. Целостность — формат (PDF/X), повреждённость, число полос, прозрачности/оверпринты.

**Три исхода — светофор, НЕ бинарно (анти-паттерн — бинарный):**
- **Зелёный** — чисто, пропускаем к оплате/цветопробе автоматически.
- **Жёлтый** — риск не блокирующий (нет вылетов — можем добавить; текст близко к краю). Предупреждаем по-человечески (с картинкой), даём осознанно продолжить. Без жаргона: не «отсутствует bleed 3mm», а «края могут обрезаться, вот как выглядит».
- **Красный** — печатать физически нельзя. Блокируем, говорим что именно и как починить.
- *Почему светофор:* каждый ложный «красный» = потерянный холодный заказ; каждый пропущенный дефект = брак за наш счёт. Светофор балансирует обе ошибки.

**Визуальный preflight** бьёт текстовый отчёт: показывать превью с наложенными линией реза, зоной вылетов, безопасной зоной. Человек видит, что объект вылезает за рез, без знания полиграфии.

**Где считается:** разбор PDF, рендер превью, анализ цветовых пространств/dpi — **серверная работа**. Это остров с тонким клиентом (загрузка, прогресс, показ результата) и серверной проверкой под ним.

**Деградация и человек в петле (обязательно):** preflight не всесилен (экзотика, хитрая прозрачность). Всегда нужен путь «не уверены? пусть проверит специалист» — это и страховка от ложных срабатываний, и мостик в B2B-трек.

**Автопочинка — с границей:** мелкое и предсказуемое чинить автоматически с уведомлением (добавить вылеты). Всё, что меняет вид (конвертация цвета RGB→CMYK, подмена шрифта) — только показывать и просить подтвердить на цветопробе. Нельзя напечатать «технически валидный, но не тот цвет».
